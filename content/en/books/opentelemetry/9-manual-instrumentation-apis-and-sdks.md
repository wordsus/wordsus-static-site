Automatic instrumentation provides a solid baseline, but it treats your application code as a black box—revealing *that* an operation occurred, but not *why*. To achieve true observability, you must embed telemetry directly into your business logic.

This chapter explores OpenTelemetry's manual instrumentation APIs. We will cover how to bootstrap the SDK and establish your telemetry pipeline. You will learn to create custom spans that map internal execution paths, record domain-specific metrics, and properly handle exceptions to distinguish expected business errors from critical system faults.

## 9.1 Initializing and Configuring the OpenTelemetry SDK

As established in Chapter 3, OpenTelemetry strictly separates its instrumentation API from its SDK. While your application code will exclusively reference the API to generate spans, metrics, and logs, those signals will simply be discarded unless an underlying SDK is instantiated and registered. Initializing the SDK is the critical first step in manual instrumentation; it acts as the central nervous system, dictating how telemetry is processed, batched, and ultimately exported to the Collector or your chosen backend.

SDK initialization must occur as early as possible in your application's lifecycle, typically in the `main` function or entry point, before any business logic or framework code executes.

### The Anatomy of SDK Initialization

Regardless of the programming language you are using, bootstrapping the OpenTelemetry SDK involves assembling a specific hierarchy of components. Understanding this hierarchy is essential for configuring a robust telemetry pipeline.

```text
[ Application Code using OTel API ]
               |
               v
+-----------------------------------+
|      Global OTel Registry         | (Connects API to SDK)
+-----------------------------------+
               |
               v
+-----------------------------------+
|          Providers                | (TracerProvider, MeterProvider)
|-----------------------------------|
| + Resource (Service metadata)     |
| + Sampler (e.g., ParentBased)     |
|                                   |
| +-- Processors (e.g., Batch)      |
|       |                           |
|       +-- Exporters (e.g., OTLP)  |
+-----------------------------------+
```

To configure the SDK, you must systematically build from the bottom up:

1.  **Define the Resource:** Identify the entity producing the telemetry.
2.  **Configure Exporters:** Determine the destination for the data.
3.  **Attach Processors:** Define how data is staged and batched before export.
4.  **Construct Providers:** Assemble the Resource, Processors, and Samplers into the engine.
5.  **Register Globally:** Bind the SDK Provider to the global API.

### Step-by-Step Configuration Strategy

Let us examine how this translates into actual code. While syntax varies across languages, the pattern remains identical. The following examples utilize Python to demonstrate the programmatic assembly of a tracing pipeline. 

#### 1. Defining the Resource
The `Resource` object represents the entity producing telemetry. Every span and metric generated by this SDK instance will be tagged with these attributes. At a minimum, you must define the `service.name`.

```python
from opentelemetry.sdk.resources import Resource, SERVICE_NAME, SERVICE_VERSION

# Define the immutable attributes of the service
resource = Resource(attributes={
    SERVICE_NAME: "checkout-service",
    SERVICE_VERSION: "1.4.2",
    "deployment.environment": "production"
})
```

#### 2. Configuring Exporters and Processors
Processors control the flow of data. For production workloads, you should always use a batch processor (e.g., `BatchSpanProcessor`) rather than a simple processor. Batching reduces network overhead and prevents the exporter from blocking the application's main execution thread. 

The processor requires an Exporter to function. Here, we configure an OTLP exporter to send data to the OpenTelemetry Collector via gRPC.

```python
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.trace.export import BatchSpanProcessor

# Configure the destination (usually an OTel Collector)
otlp_exporter = OTLPSpanExporter(endpoint="http://otel-collector:4317", insecure=True)

# Attach the exporter to a batch processor
span_processor = BatchSpanProcessor(otlp_exporter)
```

#### 3. Assembling the Provider and Global Registration
The `TracerProvider` is the factory that manages state and configuration for all generated traces. Once assembled with the Resource and Processor, it must be registered so that API calls (like `trace.get_tracer()`) route to it.

```python
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider

# 1. Initialize the Provider with the Resource
tracer_provider = TracerProvider(resource=resource)

# 2. Add the configured processor
tracer_provider.add_span_processor(span_processor)

# 3. Register the provider globally
trace.set_tracer_provider(tracer_provider)
```

### Programmatic vs. Environment Variable Configuration

While the programmatic approach shown above offers maximum flexibility—allowing you to pull configuration from databases or secret managers at runtime—it is often overly verbose for standard deployments. 

To embrace cloud-native principles, the OpenTelemetry SDKs heavily support configuration via environment variables. This allows you to write minimal initialization code and rely on the execution environment to define behavior.

If your application environment provides standard OTel variables:
* `OTEL_SERVICE_NAME="checkout-service"`
* `OTEL_TRACES_EXPORTER="otlp"`
* `OTEL_EXPORTER_OTLP_ENDPOINT="http://otel-collector:4317"`

You can often simplify your initialization sequence significantly. Most language SDKs provide auto-configuration utilities that automatically read these variables, construct the Resource, Processors, and Exporters, and wire them to the global registry in a single function call.

### Context Propagation Configuration

By default, most OpenTelemetry SDKs are configured to use the W3C Trace Context standard for propagating trace state across service boundaries. If your architecture requires interoperability with legacy systems, you must explicitly configure the global propagator during initialization.

For example, if you need to support both W3C Trace Context and AWS X-Ray headers, you would register a composite propagator:

```python
from opentelemetry import propagate
from opentelemetry.propagators.aws import AwsXRayFormat
from opentelemetry.propagators.composite import CompositePropagator
from opentelemetry.trace.propagation.tracecontext import TraceContextTextMapPropagator

# Combine W3C and X-Ray propagation protocols
propagate.set_global_textmap(
    CompositePropagator([
        TraceContextTextMapPropagator(),
        AwsXRayFormat(),
    ])
)
```

### Graceful Shutdown and Flushing

A common pitfall in manual instrumentation is the loss of telemetry data when a process terminates abruptly or completes a short-lived execution (such as a cron job or serverless function). Because production configurations rely on asynchronous batch processors, telemetry data sits in an in-memory queue waiting to be exported.

If the application exits before the batch interval triggers, the in-memory data is destroyed. To prevent this, you must hook into your application's shutdown sequence to flush the provider:

```python
import atexit

# Ensure all buffered spans are exported before the process exits
atexit.register(lambda: trace.get_tracer_provider().shutdown())
```

Invoking `shutdown()` forces the SDK to flush all remaining telemetry to the configured exporters and closes network connections cleanly. Once `shutdown()` is called, the SDK instance will drop any newly generated telemetry.

## 9.2 Creating Custom Spans and Recording Business Logic

While automatic instrumentation excels at capturing the edges of your application—such as incoming HTTP requests or outgoing database queries—it treats your application code as a black box. It knows *that* a request took 500 milliseconds, but it cannot tell you *why* the business logic inside that request took so long. 

To gain visibility into the internal workings of your application, you must use the OpenTelemetry API to create custom spans. Manual instrumentation allows you to map your telemetry directly to your business domain, capturing critical operations, user states, and domain-specific events.

### Acquiring a Tracer

Before you can create a span, you need a `Tracer`. As configured in Section 9.1, the SDK manages the global `TracerProvider`. Your application code interacts solely with the API to request a named tracer from this provider.

The name you provide to the tracer is crucial. It identifies the specific library, module, or package generating the telemetry. This makes it significantly easier to filter and route telemetry in the Collector or your backend.

```python
from opentelemetry import trace

# Acquire a tracer named after the current module or business domain
# Best practice is to include the instrumentation version
tracer = trace.get_tracer("ecommerce.checkout.processor", "1.2.0")
```

### The Span Lifecycle and Context Management

A span represents a single operation within a trace. It has a definite start time and end time. If a span is not explicitly ended, it will leak memory and never be exported.

The safest and most idiomatic way to manage a span's lifecycle in most languages is via a context manager or closure, which guarantees the span is closed even if an unhandled exception occurs within the block. When you use the "start as current span" API, OpenTelemetry automatically pushes the span into the active context.

```python
def process_checkout(cart_id: str):
    # The span starts when the block is entered and ends when exited
    with tracer.start_as_current_span("process_checkout") as span:
        # Business logic goes here
        verify_inventory(cart_id)
        calculate_totals(cart_id)
```

### Enriching Spans with Attributes

A span with just a name and duration is only marginally useful. The true power of observability comes from high-cardinality attributes. Attributes are key-value pairs that describe the state of the operation.

Unlike metrics, which break down under high cardinality (e.g., tracking millions of unique user IDs), traces are designed for it. You should liberally attach business context to your spans.

```python
def process_checkout(cart_id: str, user_id: str, customer_tier: str):
    with tracer.start_as_current_span("process_checkout") as span:
        # Attach business context
        span.set_attribute("app.cart.id", cart_id)
        span.set_attribute("app.user.id", user_id)
        span.set_attribute("app.customer.tier", customer_tier)
        
        # ... logic ...
        
        # Attributes can be updated later in the span's lifecycle
        span.set_attribute("app.checkout.total_value", 154.99)
```

**Attribute Naming Conventions:** Notice the hierarchical naming (`app.user.id`). Following the OpenTelemetry Semantic Conventions (discussed in Chapter 3) or establishing a strict internal schema prevents naming collisions and ensures your telemetry is queryable across different microservices.

### Recording Span Events

While attributes represent the state of the span as a whole, **Events** represent a point-in-time occurrence within the span. You can think of span events as structured logs that are intrinsically bound to the trace context.

Events are ideal for recording milestones within a long-running span or capturing lightweight occurrences that do not warrant the overhead of an entirely new child span.

```python
def calculate_totals(cart_id: str):
    with tracer.start_as_current_span("calculate_totals") as span:
        span.add_event("fetching_tax_rates", {"region": "US-CA"})
        # ... network call to tax service ...
        
        span.add_event("tax_rates_applied", {"tax_total": 12.50})
```

### Creating Nested Spans (Parent-Child Relationships)

Distributed tracing relies on parent-child relationships to build a causal graph of execution. When using `start_as_current_span`, OpenTelemetry automatically inspects the current execution context. If an active span already exists, the newly created span is automatically designated as its child.

By nesting your instrumented functions, you inherently build a trace tree.

```python
def process_order(order_data: dict):
    # This becomes the Root Span (or child of an incoming HTTP request)
    with tracer.start_as_current_span("process_order") as parent_span:
        parent_span.set_attribute("order.id", order_data.get("id"))
        
        # This automatically becomes a child of "process_order"
        validate_payment(order_data.get("payment_info"))
        
        # This also becomes a child of "process_order"
        dispatch_to_warehouse(order_data.get("items"))

def validate_payment(payment_info):
    with tracer.start_as_current_span("validate_payment") as child_span:
        child_span.set_attribute("payment.method", payment_info.get("type"))
        child_span.add_event("contacting_payment_gateway")
        # ... integration logic ...
```

#### Visualizing the Trace Hierarchy

The code structure above translates directly into the hierarchical trace view you will eventually see in your observability backend (such as Jaeger or Honeycomb). The SDK propagates the `Trace ID` across all spans, while managing unique `Span IDs` and `Parent Span IDs` for the relationships.

```text
Trace (ID: 5b8aa5a2...)
│
└── [Span] process_order (Span ID: 1111)
    │      Attributes: { order.id: "ORD-992" }
    │
    ├── [Span] validate_payment (Span ID: 2222, Parent ID: 1111)
    │          Attributes: { payment.method: "credit_card" }
    │          Events: [ "contacting_payment_gateway" @ t=12ms ]
    │
    └── [Span] dispatch_to_warehouse (Span ID: 3333, Parent ID: 1111)
               Attributes: { items_count: 4 }
```

By purposefully defining custom spans around critical business boundaries, you transform generic performance monitoring into a precise map of your application's behavior. *(Note: Handling failures and injecting status codes into these custom spans will be covered in Section 9.4).*

## 9.3 Registering and Recording Custom Application Metrics

While distributed traces provide deep, request-level visibility, they are inherently expensive to store and often heavily sampled (as we will explore in Chapter 16). To build reliable dashboards, set up service-level objectives (SLOs), and trigger alerts, you need aggregated, numeric time-series data. This is the domain of metrics.

Chapter 5 explored the theoretical foundations of OpenTelemetry metric instruments. In this section, we transition to practice: utilizing the OpenTelemetry API to register these instruments and record business-critical measurements from within your application code.

### Acquiring a Meter

Similar to how you must acquire a `Tracer` to create spans, you must acquire a `Meter` to create metric instruments. The `Meter` is instantiated by the global `MeterProvider` (configured during SDK initialization) and should be named according to the logical component generating the metrics.

```python
from opentelemetry import metrics

# Acquire a meter named after the business domain
meter = metrics.get_meter("ecommerce.checkout.metrics", "1.2.0")
```

Once you have a meter, you can create various instruments. It is best practice to declare your instruments at the module or class level during application startup, rather than recreating them on the fly for every request.

### Synchronous Instruments

Synchronous instruments are called manually by the application code precisely when an event occurs. They are deeply embedded within your business logic.

#### 1. The Counter
A `Counter` is a monotonic instrument, meaning it only goes up. It is ideal for tracking occurrences, such as the total number of processed orders, emails sent, or API errors. 

```python
# 1. Register the instrument
items_sold_counter = meter.create_counter(
    name="ecommerce.checkout.items_sold",
    description="Total number of individual items processed through checkout",
    unit="{item}"
)

def process_cart(cart):
    # ... checkout logic ...
    
    # 2. Record a measurement synchronously
    # Attributes provide the dimensions for grouping in your dashboard
    items_sold_counter.add(len(cart.items), {
        "payment.method": cart.payment_type,
        "customer.tier": cart.customer_tier
    })
```

#### 2. The UpDownCounter
An `UpDownCounter` functions like a standard counter but allows negative increments. It is designed to track resources that grow and shrink over time, such as the number of items currently sitting in active shopping carts.

```python
active_carts_counter = meter.create_up_down_counter(
    name="ecommerce.checkout.active_carts",
    description="Number of carts currently being actively modified",
)

def create_cart():
    active_carts_counter.add(1)

def abandon_or_checkout_cart():
    active_carts_counter.add(-1)
```

#### 3. The Histogram
A `Histogram` captures a distribution of values. It is the workhorse for measuring latencies, payload sizes, or financial values. The OpenTelemetry SDK automatically buckets these measurements, allowing your backend to calculate percentiles (e.g., P95, P99) without retaining every individual data point.

```python
checkout_value_histogram = meter.create_histogram(
    name="ecommerce.checkout.order_value",
    description="Distribution of final order totals in USD",
    unit="USD"
)

def process_checkout(cart):
    final_total = calculate_total(cart)
    # ... logic ...
    
    checkout_value_histogram.record(final_total, {
        "region": cart.shipping_region
    })
```

### Asynchronous Instruments (Observables)

While synchronous instruments are driven by application events, asynchronous (observable) instruments are driven by the OpenTelemetry SDK's collection cycle. When the metric exporter runs (e.g., every 60 seconds), it invokes a registered callback function to fetch the current value.

Asynchronous instruments are perfect for expensive operations or observing state that is not directly tied to a specific request, such as thread pool utilization, queue depth, or memory usage.

```python
from typing import Iterable
from opentelemetry.metrics import CallbackOptions, Observation

# The callback function must yield Observation objects
def get_payment_queue_depth(options: CallbackOptions) -> Iterable[Observation]:
    # Imagine this makes a network call to RabbitMQ or Redis
    current_depth = payment_queue.get_message_count()
    
    yield Observation(current_depth, {"queue.name": "payment_processing"})

# Register the callback with an Observable Gauge
meter.create_observable_gauge(
    name="ecommerce.checkout.payment_queue_depth",
    callbacks=[get_payment_queue_depth],
    description="Current number of messages waiting in the payment queue",
    unit="{message}"
)
```

### The Cardinality Warning: Attributes in Metrics vs. Traces

When recording custom business logic, it is incredibly tempting to use the same rich context for metrics that you use for traces. **Do not do this.** There is a fundamental architectural difference between trace attributes and metric attributes:

* **Traces (High Cardinality Safe):** You *should* attach `order_id="12345"` or `user_id="U-998"` to a span. The tracing backend stores these as individual, searchable events.
* **Metrics (High Cardinality Danger):** Every unique combination of metric attributes generates a new, distinct time-series in memory and in your metrics database (like Prometheus). 

```text
# DANGEROUS METRIC USAGE:
items_sold_counter.add(1, {"order_id": "ORD-1001"}) # Creates Timeseries A
items_sold_counter.add(1, {"order_id": "ORD-1002"}) # Creates Timeseries B
items_sold_counter.add(1, {"order_id": "ORD-1003"}) # Creates Timeseries C
```

If your e-commerce site processes 50,000 orders a day, attaching `order_id` to a metric will generate 50,000 distinct time-series. This is known as "cardinality explosion" and will crash the OpenTelemetry Collector, incur massive vendor bills, or bring down your observability backend via Out-Of-Memory (OOM) errors.

**Rule of Thumb for Metric Attributes:** Only use attributes with a small, bounded set of possible values (e.g., `HTTP status codes`, `regions`, `customer_tiers`, `payment_methods`). If a value can grow infinitely, it belongs on a trace or a log, not a metric.

## 9.4 Proper Error Handling and Status Code Injection

A trace that simply records latency and attributes tells only half the story. When an application inevitably fails, observability tools must rapidly surface the failure, pinpoint its location in the distributed architecture, and provide the contextual evidence needed to debug it. 

In OpenTelemetry, error tracking is not a separate signal; it is an intrinsic property of the Span lifecycle. However, effectively instrumenting errors requires a nuanced understanding of how to differentiate between expected business rule violations and unexpected system faults.

### The Span Status Model

Every OpenTelemetry span possesses a `Status` property. This is the primary mechanism your observability backend uses to calculate error rates, trigger alerts, and color-code failed operations (typically in red) within a trace waterfall visualization.

OpenTelemetry defines three explicit states for a span's status:

1.  **`UNSET` (Default):** The operation completed without explicitly declaring success or failure. Most backend systems treat this as a successful execution by default.
2.  **`OK`:** The developer explicitly marked the operation as successful. This is rarely necessary unless you are specifically overriding an automated instrumentation that previously flagged an error.
3.  **`ERROR`:** The operation encountered a definitive, system-level failure.

### Recording Exceptions and Setting Status

To properly capture an error during manual instrumentation, you must perform two distinct actions within your error handling logic: record the exception details, and update the span's status.

```text
[ Application Exception Thrown ]
              |
              v
+-----------------------------+
|    try / except Block       |
+-----------------------------+
      |                 |
      v                 v
[ record_exception() ] [ set_status(ERROR) ]
(Extracts Stacktrace)  (Flags Span as Failed)
      |                 |
      +--------+--------+
               |
               v
  [ Exported to Backend ]
```

The `record_exception()` API is a convenience method that automatically extracts the exception type, the error message, and the full stack trace. It formats this data and attaches it to the span as a standard Span Event following OpenTelemetry Semantic Conventions (e.g., `exception.type`, `exception.stacktrace`). 

However, **recording an exception does not automatically change the span's status to `ERROR`.** You must explicitly mutate the status using `set_status()`.

#### Python Implementation Example

```python
from opentelemetry.trace.status import Status, StatusCode

def process_payment(payment_payload: dict):
    with tracer.start_as_current_span("process_payment") as span:
        try:
            # Attempt a risky network call
            charge_credit_card(payment_payload)
            
        except ConnectionError as e:
            # 1. Record the stack trace and exception details as an event
            span.record_exception(e)
            
            # 2. Explicitly mark the span as failed
            span.set_status(
                Status(StatusCode.ERROR, description="Payment gateway timeout")
            )
            
            # 3. Handle the error or re-raise it appropriately
            raise
```

### The Nuance: Business Errors vs. System Faults

A common anti-pattern in manual observability is over-instrumenting errors. This leads to "alert fatigue," where engineers stop trusting their dashboards because the system constantly reports false positives. 

You must draw a strict line between **System Faults** and **Business Errors**.

* **System Fault:** A database connection drops, a downstream microservice times out, or a `NullReferenceException` occurs. These represent a failure of the software to operate as designed. **Action:** Record the exception and set the span status to `ERROR`.
* **Business Error:** A user submits an invalid password, a requested item is out of stock, or an API returns a `400 Bad Request` due to malformed input. The software operated exactly as designed by catching the bad input and rejecting it. **Action:** Do *not* set the span status to `ERROR`.

#### Handling Business Errors Correctly

If an end-user attempts to log in with an incorrect password, the authentication service has technically succeeded in its job (denying access). If you mark this span as an `ERROR`, your service's error rate will spike every time a user forgets their password, triggering pager alerts at 3:00 AM.

Instead of modifying the span status, use Span Attributes or custom Metrics to track business-level rejections:

```python
def authenticate_user(username, password):
    with tracer.start_as_current_span("authenticate_user") as span:
        span.set_attribute("app.user.login", username)
        
        is_valid = verify_credentials(username, password)
        
        if not is_valid:
            # Log the business failure via attributes, keeping status UNSET/OK
            span.set_attribute("app.auth.success", False)
            span.set_attribute("app.auth.failure_reason", "invalid_password")
            
            # Increment a metric for security monitoring
            login_failures_counter.add(1, {"reason": "invalid_password"})
            return False
            
        span.set_attribute("app.auth.success", True)
        return True
```

By keeping the span status `UNSET` while adding descriptive attributes, you ensure your SRE team only wakes up for actual outages, while your product and security teams can still query the backend to analyze user-facing friction and failed login attempts.
