While distributed traces reveal the intricate journey of individual requests, they are inherently microscopic. To understand the macroeconomic health of a system—its overall performance, resource utilization, and error rates over time—we need a different telemetry signal: Metrics. Metrics provide quantitative, aggregated measurements of system behavior, optimized for rapid querying, alerting, and efficient long-term storage. 

In this chapter, we explore the OpenTelemetry metric data model. We will examine the distinct instruments used to capture measurements, the mechanics of synchronous and asynchronous data collection, the critical concept of aggregation temporality, and how to seamlessly bridge high-level metric anomalies with low-level trace data using exemplars.

## 5.1 Exploring Metric Instruments (Counters, Gauges, Histograms)

In OpenTelemetry, telemetry data is generated by interacting with the API. For metrics, this API is exposed through a `Meter`, which in turn creates **Instruments**. An instrument is the specific logical device you use to record measurements about your application's execution.

To capture the physical realities of different systems—from tracking the number of HTTP requests, to measuring the current memory footprint, to recording the latency of database queries—OpenTelemetry provides several distinct types of metric instruments. These instruments are categorized along two primary dimensions: **Synchronicity** and **Semantic Behavior** (Counters, Gauges, and Histograms).

### The Dimension of Synchronicity

Before diving into the specific instruments, it is crucial to understand the execution model of how measurements are recorded, as this dictates which instrument you should choose:

1.  **Synchronous Instruments:** These are called inline with your application's business logic. Because they are recorded during the execution of a request or operation, they are inherently tied to the active OpenTelemetry `Context`. This allows synchronous measurements to automatically associate with the current Trace ID and Span ID (a concept we will expand upon in Section 5.5 with Exemplars).
2.  **Asynchronous (Observable) Instruments:** These are invoked via a callback registered with the OpenTelemetry SDK. The SDK periodically calls this function in the background during a collection cycle. Asynchronous instruments are ideal for capturing the state of a system (e.g., garbage collection statistics, CPU load) where the measurement is completely independent of any specific user request. 

### Counters: Measuring Accumulation

Counters are used to measure additive values. They record discrete events or quantities that accumulate over time.

#### 1. Counter (Synchronous)
The standard Counter is **monotonically increasing**. This means it can only go up, never down (or record negative values). It is the most common instrument used to calculate the *rate* of events.

* **Use Cases:** Number of HTTP requests processed, bytes sent over a network, number of orders completed, exceptions thrown.
* **Behavior:** You call `.add(value)` where `value >= 0`.

```python
# Python Example: Using a Synchronous Counter
from opentelemetry import metrics

meter = metrics.get_meter("my.ecommerce.service")
order_counter = meter.create_counter(
    name="ecommerce.orders.total",
    description="Total number of processed orders",
    unit="{order}"
)

def process_order(order_payload):
    # Business logic here...
    
    # Record the measurement inline with the operation
    order_counter.add(1, {"order.type": "digital", "customer.tier": "premium"})
```

#### 2. UpDownCounter (Synchronous)
The `UpDownCounter` is identical to a standard Counter, but it is **non-monotonic**. It allows for negative increments, meaning the accumulated value can fluctuate up and down. 

* **Use Cases:** Number of active concurrent requests, items currently in a queue, active database connections.
* **Behavior:** You call `.add(value)` where `value` can be positive or negative.

#### 3. Observable Counters (Asynchronous)
OpenTelemetry provides asynchronous equivalents: `ObservableCounter` and `ObservableUpDownCounter`. Instead of calling `.add()` inline, you provide a callback that reports the *absolute total* when polled. This is particularly useful when you are reading metrics from an external system that already maintains the cumulative count (e.g., reading network bytes transmitted directly from the Linux `/proc/net/dev` file).

### Histograms: Measuring Distributions

A `Histogram` is a synchronous instrument used to record a distribution of values. Unlike a Counter that just sums up a total, a Histogram groups recorded values into buckets (or calculates percentiles) to help you understand the statistical distribution of the data. 

* **Use Cases:** Request latency, payload sizes, processing time.
* **Behavior:** You call `.record(value)`. The OpenTelemetry SDK then aggregates these individual measurements into statistical summaries (min, max, sum, count, and bucket counts).

Histograms are critical for observability because averages (means) lie. If 99 requests take 1ms and 1 request takes 1000ms, the average is ~11ms, hiding the fact that one user had a terrible experience. Histograms allow you to visualize the 95th or 99th percentiles (p95, p99).

```go
// Go Example: Using a Histogram
package main

import (
	"context"
	"time"
	"go.opentelemetry.io/otel/metric"
)

func handleRequest(ctx context.Context, meter metric.Meter) {
	// Create the histogram
	latencyHistogram, _ := meter.Float64Histogram(
		"http.server.request.duration",
		metric.WithDescription("Duration of HTTP requests"),
		metric.WithUnit("s"),
	)

	startTime := time.Now()
	
	// ... simulate work ...
	time.Sleep(50 * time.Millisecond) 

	// Record the duration as a float in seconds
	duration := time.Since(startTime).Seconds()
	latencyHistogram.Record(ctx, duration)
}
```

### Gauges: Measuring Current State

A Gauge represents a non-additive value for which the current state is the only thing that matters. Unlike Counters, where you want to know the *rate* of change, or Histograms, where you want a *distribution* of events, Gauges simply tell you "what is the value right now?"

#### ObservableGauge (Asynchronous)
The most common implementation of a gauge is the asynchronous `ObservableGauge`. Because state like CPU or memory isn't tied to a specific request, it makes sense to poll it on a scheduled interval.

* **Use Cases:** CPU utilization, memory heap usage, room temperature, disk space available.

```python
# Python Example: Using an Observable Gauge
from opentelemetry import metrics
import psutil

meter = metrics.get_meter("system.metrics")

def get_cpu_usage(options: metrics.CallbackOptions):
    # Yields the current CPU percentage when the SDK collects metrics
    yield metrics.Observation(psutil.cpu_percent(), {"cpu.id": "0"})

# Register the callback with the ObservableGauge
meter.create_observable_gauge(
    name="system.cpu.utilization",
    callbacks=[get_cpu_usage],
    description="Current CPU utilization percentage",
    unit="1"
)
```

*(Note: While the OpenTelemetry specification recently introduced a synchronous `Gauge` for rare cases where a non-additive measurement is specifically tied to an application event, the `ObservableGauge` remains the standard for the vast majority of state-measurement scenarios.)*

### Instrument Decision Tree

Choosing the correct instrument is critical, as it dictates how the SDK aggregates the data and how your backend visualizes it. Use the following logic to select the appropriate instrument:

```text
+-----------------------------------------------------------------------+
|                       METRIC INSTRUMENT DECISION TREE                 |
+-----------------------------------------------------------------------+
                            
[Are you measuring a distribution of values, like latency or size?]
  |
  +-- YES --> HISTOGRAM (Sync)
  |
  +-- NO ---> [Is the measurement recorded as part of an active request/event?]
                |
                +-- YES (Synchronous) ---> [Do you want to record a delta/change?]
                |                            |
                |                            +-- YES -> [Can it go down?]
                |                            |            |-- YES: UPDOWNCOUNTER
                |                            |            |-- NO:  COUNTER
                |                            |
                |                            +-- NO --> GAUGE (Sync) 
                |                                       (Records absolute value)
                |
                +-- NO (Asynchronous) ---> [What does the callback report?]
                                             |
                                             +-- Current State / Level
                                             |     --> OBSERVABLE GAUGE
                                             |
                                             +-- Accumulated Total (Monotonic)
                                             |     --> OBSERVABLE COUNTER
                                             |
                                             +-- Accumulated Total (Can decrease)
                                                   --> OBSERVABLE UPDOWNCOUNTER
```

By strictly adhering to these semantic instruments, you ensure that the metrics emitted by your services can be properly parsed, aggregated, and visualized by any compliant observability backend. In the next section, we will look closer at the implications of when and how these metrics are exported by examining Aggregation Temporality.

## 5.2 Synchronous vs. Asynchronous Metric Collection

While Chapter 5.1 introduced the logical instruments available in OpenTelemetry, understanding *how* the OpenTelemetry SDK collects and processes data from these instruments is equally important. The collection mechanism directly impacts application performance, data freshness, and the ability to correlate metrics with distributed traces. 

OpenTelemetry divides metric collection into two fundamental paradigms: **Synchronous** and **Asynchronous**.

### Synchronous Collection: The "Push" Model

Synchronous metric collection occurs inline with your application's active execution path. When your code processes a request, it explicitly invokes the metric instrument (e.g., calling `counter.add(1)` or `histogram.record(latency)`). You are effectively "pushing" the measurement into the OpenTelemetry SDK at the exact moment the event occurs.

#### Key Characteristics of Synchronous Collection

* **Context Awareness:** Because the measurement is recorded during the execution of a specific operation, the OpenTelemetry SDK can automatically access the active `Context`. This means synchronous metrics are implicitly linked to the current Trace ID and Span ID, allowing for powerful correlation features like Exemplars (covered in Section 5.5).
* **Event-Driven Burstiness:** Data is generated only when events happen. If a service receives a sudden spike in traffic, the SDK will receive a corresponding spike in synchronous metric calls.
* **Performance Considerations:** Since these calls happen on the critical path of your business logic, the OpenTelemetry SDK is highly optimized to ensure they are non-blocking and computationally inexpensive. Under the hood, the SDK typically updates atomic counters or lock-free data structures in memory, deferring the heavy lifting of aggregation and exportation to a background thread.

#### Visualizing the Synchronous Flow

```text
[ Application Thread ]                     [ OpenTelemetry SDK Background Thread ]

 1. Process Request
 2. Do Business Logic
 3. counter.add(1) ------(Updates State)-----> In-Memory Aggregator
 4. Return Response                                     |
                                                        | (Scheduled Interval)
                                                        v
                                               Metric Export Pipeline
```

### Asynchronous Collection: The "Pull" Model

Asynchronous metric collection decouples the measurement process from the application's request lifecycle. Instead of recording data inline, you register a callback function with the SDK during the application's startup phase. 

The SDK acts as an orchestrator, periodically "pulling" data by invoking this callback on a background thread during its scheduled metric collection cycle (e.g., every 60 seconds).

#### Key Characteristics of Asynchronous Collection

* **Context Independence:** Callbacks run in a background thread managed by the SDK, entirely separate from any user request. Therefore, asynchronous metrics have **no connection** to active traces or spans.
* **Predictable Overhead:** The frequency of data collection is strictly controlled by the SDK's export interval, regardless of application throughput. Whether your service handles 10 requests per second or 10,000, the asynchronous callback fires at the exact same rate.
* **Ideal for External State:** This model is perfect for measuring state that exists outside of a specific transaction footprint. Examples include reading the size of a connection pool, scraping garbage collection statistics from the JVM, or querying a database table for a total row count.

#### Visualizing the Asynchronous Flow

```text
[ Application Thread ]                     [ OpenTelemetry SDK Background Thread ]

 1. App Startup                                         
 2. Register Callback ------------------------> Callback Registry
 3. Serve Traffic                                       |
    (No metric calls)                                   | (Collection Cycle Trigger)
                                                        v
                                                Execute Callback
                                                        |
                                       <------(Yields Observations)
                                                        |
                                                In-Memory Aggregator
                                                        |
                                                Metric Export Pipeline
```

### Implementing the Paradigms in Code

To illustrate the architectural difference, consider how a Java application might measure the number of processed messages (Synchronous) versus the size of the message queue (Asynchronous).

```java
// Java OpenTelemetry Example

// 1. SYNCHRONOUS: Pushing data inline
LongCounter messagesProcessed = meter.counterBuilder("messages.processed.total")
    .setDescription("Total messages processed inline")
    .build();

public void processMessage(Message msg) {
    // ... logic to process message ...
    
    // Recorded exactly when the event happens, on the application thread.
    // Has access to the current Trace Context.
    messagesProcessed.add(1); 
}


// 2. ASYNCHRONOUS: Registering a callback for the SDK to pull
Queue<Message> messageQueue = getApplicationQueue();

meter.gaugeBuilder("message.queue.size")
    .ofLongs()
    .setDescription("Current size of the message queue")
    // The callback is registered once. The SDK will invoke it periodically.
    .buildWithCallback(observableMeasurement -> {
        // This runs on the SDK's background thread.
        // No Trace Context is available here.
        long currentSize = messageQueue.size();
        observableMeasurement.record(currentSize);
    });
```

### Comparison Matrix

Choosing the right collection model is just as critical as choosing the right instrument. Use the following table to summarize the architectural boundaries:

| Feature | Synchronous Collection | Asynchronous Collection |
| :--- | :--- | :--- |
| **Execution Trigger** | Application logic (Event-driven) | SDK Timer (Time-driven) |
| **Thread Context** | Application's request thread | SDK's background thread |
| **Trace Correlation** | **Yes** (Inherits active Context) | **No** (Context is empty) |
| **Data Freshness** | Immediate / Real-time | Bound by the export interval |
| **Typical Instruments** | `Counter`, `UpDownCounter`, `Histogram` | `ObservableGauge`, `ObservableCounter` |
| **Primary Use Case** | Request latency, error rates, bytes sent | CPU usage, queue depth, cache size |

### Advanced Architectural Considerations

When architecting systems heavily reliant on asynchronous metrics, engineers must account for callback performance. Because the SDK invokes all registered callbacks during its collection cycle, a slow callback (e.g., one that makes a sluggish external API call or runs a complex database query) will stall the entire metric export pipeline. 

To mitigate this, robust implementations often decouple the observation from the external system. Instead of the OpenTelemetry callback querying a database directly, a separate application cron job queries the database and caches the result in a local variable. The OpenTelemetry callback then simply reads that local variable in constant time, ensuring the metric collection cycle remains lightweight and uninterrupted.

## 5.3 Understanding Aggregation Temporality (Delta vs. Cumulative)

When an OpenTelemetry SDK collects metric data—whether synchronously inline or asynchronously via a callback—it must decide how to package that data for export. This decision is governed by **Aggregation Temporality**, which defines the time window over which metric data is aggregated. 

Put simply, temporality answers the question: *Are we reporting the total value since the application started, or only the change in value since the last time we reported?*

In OpenTelemetry, there are two primary temporality models: **Cumulative** and **Delta**. Understanding the difference is crucial, as choosing the wrong temporality for your specific observability backend will result in wildly inaccurate dashboards.

*(Note: Temporality only applies to instruments that aggregate data over time, such as Counters and Histograms. Gauges, which report a point-in-time state, do not have temporality.)*

### The Core Difference

To visualize the difference, imagine an e-commerce service processing orders. In a 40-second window, orders come in bursts. We configure the SDK to export metrics every 10 seconds.

```text
+-----------------------------------------------------------------------+
|                 VISUALIZING METRIC TEMPORALITY                        |
+-----------------------------------------------------------------------+
Time Window:       [0s - 10s]   [10s - 20s]   [20s - 30s]   [30s - 40s]

Orders Processed:   +5 orders    +3 orders     0 orders      +8 orders

-------------------------------------------------------------------------
CUMULATIVE EXPORT:      5            8            8             16
(Total since start)

DELTA EXPORT:           5            3            0             8
(Change since last)
-------------------------------------------------------------------------
```

### Cumulative Temporality

In the Cumulative model, the SDK maintains the state of the aggregation from the moment the application process starts (or the instrument is created). Every time the SDK exports data, it sends the grand total up to that point. 

* **How Backends Use It:** The observability backend (like Prometheus) is responsible for calculating rates. To figure out how many orders happened between 10s and 20s, the backend subtracts the value at T1 (5) from the value at T2 (8) to get the rate of 3.
* **Resilience to Data Loss:** Cumulative temporality is highly resilient to transient network failures. If the export at the 20-second mark fails and the packet is dropped, the data isn't lost. When the 30-second export succeeds (reporting 8), the backend can simply calculate the rate over a 20-second window instead of a 10-second window.
* **The "Counter Reset" Problem:** If the application crashes and restarts, the cumulative counter resets to 0. Observability backends must be smart enough to detect this reset (usually by noticing the new value is lower than the previous value) and handle the math gracefully.

### Delta Temporality

In the Delta model, the SDK resets its internal aggregator back to zero immediately after successfully exporting the data. Every export represents strictly the exact volume or distribution of events that occurred *only* during that specific export interval.

* **How Backends Use It:** Backends (like StatsD or Datadog) simply sum up the incoming delta values over whatever time window the user is viewing on their dashboard.
* **Vulnerability to Data Loss:** Delta temporality is fragile over unreliable networks. If the export at the 40-second mark (reporting 8 orders) is dropped by the network, those 8 orders are permanently lost. The backend has no way to know they ever existed, resulting in an undercount.
* **Memory Efficiency:** For high-cardinality Histograms, Delta temporality can sometimes save memory in the SDK, as unused buckets reset to zero and don't need to be continually tracked and exported.

### Choosing Between Delta and Cumulative

OpenTelemetry supports both models because the broader observability ecosystem is split on the preferred approach. Your choice of temporality is almost entirely dictated by the backend system you are sending the data to.

1.  **Prometheus Ecosystem:** Prometheus is fundamentally designed around **Cumulative** metrics. It expects to pull grand totals and uses functions like `rate()` to calculate the delta on the server side. Sending Delta metrics to Prometheus will result in broken graphs.
2.  **StatsD / Commercial APMs:** Many traditional monitoring tools and commercial vendors (e.g., Datadog, Dynatrace) historically prefer **Delta** metrics for counters, as it simplifies their backend storage and aggregation logic.

### Configuring Temporality in the SDK

By default, most OpenTelemetry SDKs prefer Cumulative temporality to ensure data resilience. However, you can configure the SDK to output Delta metrics. This is typically done at the `MetricReader` or `Exporter` level, rather than on the individual instrument.

Here is an example of how you might configure a Delta preference when setting up an OTLP exporter in Python:

```python
from opentelemetry import metrics
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.metrics.export import (
    PeriodicExportingMetricReader,
    AggregationTemporality,
)
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter

# Define a function that dictates the preferred temporality
def temporality_selector(instrument_type):
    # We want to export our Counter and Histogram data as Deltas
    return AggregationTemporality.DELTA

# Initialize the Exporter with the temporality selector
exporter = OTLPMetricExporter(
    endpoint="http://my-collector:4317",
    preferred_temporality=temporality_selector
)

# Attach the exporter to the reader
reader = PeriodicExportingMetricReader(exporter)

# Initialize the global MeterProvider
provider = MeterProvider(metric_readers=[reader])
metrics.set_meter_provider(provider)
```

In a production environment, you will often decouple this decision from your application code. The application will export the SDK defaults (Cumulative) via OTLP to an OpenTelemetry Collector. The Collector can then be configured to convert Cumulative metrics to Delta metrics before forwarding them to a backend that requires them, ensuring your application code remains vendor-agnostic.

## 5.4 Configuring Views and Metric Pipelines

Up to this point, we have explored how developers create instruments to generate telemetry and how the SDK calculates temporality over time. However, a major challenge in enterprise observability is the disconnect between *what developers instrument* and *what operators actually want to store*. 

Developers tend to over-instrument, capturing high-cardinality attributes (like user IDs or raw URLs) and logging every minor state change. If left unchecked, this raw data flows directly into the observability backend, leading to massive storage costs and degraded query performance. 

OpenTelemetry solves this tension through **Metric Pipelines** and the **View API**. Views provide a configuration layer that allows operators to modify, filter, or completely drop metrics before they are aggregated and exported, entirely without modifying the original application source code.

### The Anatomy of a Metric Pipeline

A Metric Pipeline is the journey a measurement takes from the moment `instrument.record()` is called to the moment it leaves the process via network export. 

```text
+-------------------+       +-------------------+       +-------------------+
|  Instrumentation  |       |   SDK Processing  |       |     Exporting     |
+-------------------+       +-------------------+       +-------------------+
|                   |       |                   |       |                   |
|  Meter.counter()  | ====> |     [ View ]      | ====> |  MetricReader     |
|  Meter.histogram()|       |  (Filter/Modify)  |       |  (Interval pull)  |
|                   |       |         |         |       |         |         |
+-------------------+       |         v         |       |         v         |
                            |   Aggregator      |       |  OTLP Exporter    |
                            +-------------------+       +-------------------+
```

1.  **The Instrument:** The logical entry point in the application code.
2.  **The View:** The routing and mutation rules applied to incoming measurements.
3.  **The Aggregator:** The in-memory state engine that groups data (e.g., summing a counter, binning a histogram) based on the View's instructions.
4.  **The MetricReader:** The engine that orchestrates the collection cycle, polling the Aggregators for their current state.
5.  **The Exporter:** The protocol-specific component (usually OTLP) that serializes the data and sends it over the network.

### The Power of the View API

The View API is the most powerful tool for controlling metric costs and shaping data for specific Service Level Objectives (SLOs). A View targets specific instruments (using wildcard matching on metric names, meter names, or instrument types) and applies one of three primary transformations:

#### 1. Attribute Filtering (Controlling Cardinality)
Every unique combination of attributes attached to a metric creates a new time series. If a developer attaches an `http.client_ip` attribute to a request counter, the application will generate millions of unique time series, causing an immediate cardinality explosion. 

A View can be configured to drop specific attributes, aggregating the data up to a safer, lower-cardinality level before it is ever stored in memory.

#### 2. Modifying Aggregations (Custom Histograms)
By default, OpenTelemetry SDKs use standard bucket boundaries for Histograms (e.g., `[0, 5, 10, 25, 50, 75, 100...]`). However, if your service has a strict SLO stating that 99% of requests must complete in under 12 milliseconds, the default buckets (jumping from 10 to 25) are useless. You can use a View to override the default aggregation and define explicit bucket boundaries tailored to your business requirements.

#### 3. Dropping Metrics entirely
If an external library introduces noisy, unhelpful metrics, or if a legacy metric is no longer needed but the application code hasn't been updated yet, a View can instruct the SDK to use a `Drop` aggregation. The SDK will completely ignore measurements for that instrument, costing zero memory and zero network bandwidth.

### Implementing Views in the SDK

Views are registered when initializing the `MeterProvider`. The order of registration matters; the SDK evaluates instruments against Views in the order they are defined. If an instrument matches multiple Views, it can output multiple metric streams (e.g., one standard metric, and one specialized metric with different histogram buckets).

Here is an example demonstrating these three core View capabilities using the OpenTelemetry Python SDK:

```python
from opentelemetry import metrics
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader, ConsoleMetricExporter
from opentelemetry.sdk.metrics.view import View, DropAggregation, ExplicitBucketHistogramAggregation

# 1. DROP VIEW: Completely ignore noisy database metrics from a 3rd party library
drop_noisy_metrics = View(
    instrument_name="db.connection.pool.*", # Wildcard matching
    aggregation=DropAggregation()
)

# 2. ATTRIBUTE FILTER VIEW: Prevent cardinality explosion on HTTP counters
# We only want to keep the 'http.status_code' and 'http.method' attributes.
# If the developer logged 'user_id' or 'client_ip', they will be silently dropped.
filter_attributes = View(
    instrument_name="http.server.requests",
    allowed_attribute_keys={"http.status_code", "http.method"}
)

# 3. AGGREGATION VIEW: Custom histogram buckets for latency SLOs
# We care deeply about sub-50ms performance, so we define tight buckets there.
custom_histogram_buckets = View(
    instrument_name="http.server.request.duration",
    aggregation=ExplicitBucketHistogramAggregation(
        boundaries=(1.0, 5.0, 12.0, 25.0, 50.0, 100.0, 500.0)
    )
)

# Initialize the Pipeline
reader = PeriodicExportingMetricReader(ConsoleMetricExporter())
provider = MeterProvider(
    metric_readers=[reader],
    views=[drop_noisy_metrics, filter_attributes, custom_histogram_buckets]
)

metrics.set_meter_provider(provider)
```

### Best Practices for Metric Pipelines

* **Shift Left on Filtering:** While you can filter metrics downstream in the OpenTelemetry Collector (as we will see in Chapter 14), it is significantly more efficient to use SDK Views to drop high-cardinality data. If you wait until the Collector, your application SDK still has to waste CPU and RAM maintaining those massive aggregations in memory, risking an Out-Of-Memory (OOM) crash in your application containers.
* **Decouple Configuration:** Hardcoding Views into application logic defeats their purpose. In mature enterprise deployments, operators externalize View configurations via environment variables or configuration files that the application parses at startup. This allows infrastructure teams to dynamically tweak metric pipelines without requiring developers to redeploy code.
* **Use Exponential Histograms:** If manually configuring explicit bucket boundaries via Views becomes tedious across hundreds of services, consider configuring your Views to use `Base2ExponentialHistogramAggregation`. This automatically scales the resolution of buckets based on the data distribution, eliminating the need to manually guess the correct boundaries for latency distributions.

## 5.5 Using Exemplars to Bridge Metrics and Traces

A fundamental axiom of observability is that metrics and traces serve different, complementary purposes. Metrics are cheap to store and excellent at alerting you that **a problem exists** (e.g., "The 99th percentile latency of the checkout service just spiked to 4 seconds"). Traces, on the other hand, are rich in detail and excellent at telling you **why the problem exists** (e.g., "The database span took 3.9 seconds due to a missing index").

However, a massive gap exists between these two signals. When your dashboard shows a p99 latency spike, how do you find the *exact* distributed trace that caused it among the millions of requests processed in that minute? Searching blindly through your tracing backend for "requests taking exactly 4 seconds" is inefficient and often inconclusive.

OpenTelemetry solves this "needle in the haystack" problem using **Exemplars**.

### What is an Exemplar?

An exemplar is a specific, representative measurement extracted from a larger aggregated metric, deeply enriched with context. Practically speaking, when the OpenTelemetry SDK aggregates a synchronous metric (like adding a latency float to a Histogram bucket), it can optionally attach the active **Trace ID** and **Span ID** to that specific data point. 

Instead of just knowing that *some* request took 4 seconds, the observability backend receives the metric data alongside a pointer: *"Here is the exact Trace ID of a request that took 4 seconds during this time window."*

### Visualizing the Exemplar Bridge

Consider a latency histogram visualized in a dashboard. The bars represent the distribution of thousands of requests. The exemplars (represented as stars) are specific, recorded instances within those buckets.

```text
+-----------------------------------------------------------------------+
|                 LATENCY HISTOGRAM WITH EXEMPLARS                      |
+-----------------------------------------------------------------------+
Count
 500 |        [   ]
 400 |        [   ]
 300 |    [   ][   ]
 200 |    [   ][   ][   ]                         * <-- Exemplar 2
 100 |    [   ][   ][   ][   ]                   [   ] (TraceID: f8e9d0...)
   0 +----[___][___][___][___]-------------------[___]-------------------
         10ms  50ms 100ms 200ms       ...        5.0s
           ^
           |
           * <-- Exemplar 1 (TraceID: a1b2c3...)
```

In a modern observability UI (like Grafana integrated with Prometheus and Tempo), a user investigating the spike can simply click the star over the 5.0s bucket. The UI reads the Exemplar's Trace ID and immediately transitions the user to the exact distributed trace waterfall that caused the anomaly, bridging the gap between aggregate statistics and individual request execution in a single click.

### How Exemplars Work Under the Hood

Because exemplars require access to the active execution context (the Trace ID), they can **only be recorded by Synchronous instruments** (Counters, UpDownCounters, and Histograms). Asynchronous instruments (Observable Gauges) run on a background thread and have no associated trace context, rendering them incapable of generating exemplars.

#### The Exemplar Reservoir and Sampling

Attaching a Trace ID to every single metric increment would defeat the purpose of metrics, causing severe memory overhead and massive payload sizes. To mitigate this, OpenTelemetry uses an **Exemplar Reservoir**. 

The reservoir is an algorithmic holding area in the SDK's memory that decides which specific measurements to keep as exemplars and which to discard. For Histograms, the standard behavior is to keep a reservoir *per bucket*. When the collection cycle occurs, the SDK exports the aggregated metric along with the small subset of retained exemplars.

You can configure how the SDK samples exemplars using an **Exemplar Filter**. The three common filters are:
1.  **AlwaysOn:** Samples all measurements (useful only in low-throughput dev environments).
2.  **AlwaysOff:** Disables exemplars entirely to save memory.
3.  **TraceBased (Default in most SDKs):** The SDK will only consider recording an exemplar if the current active trace is *sampled* (i.e., it is scheduled to be exported to the tracing backend). This prevents the SDK from exporting an exemplar pointing to a Trace ID that was dropped by your trace sampling rules, which would result in a frustrating dead link in your UI.

### Configuring Exemplars in the SDK

While many OpenTelemetry SDKs enable trace-based exemplars by default, high-performance environments often require explicit configuration of the reservoir and filter mechanisms. 

Here is how you might configure a custom Exemplar Filter in the Java SDK to explicitly ensure only sampled traces are used as exemplars:

```java
// Java Example: Configuring Exemplar Filters
import io.opentelemetry.sdk.metrics.SdkMeterProvider;
import io.opentelemetry.sdk.metrics.internal.exemplar.ExemplarFilter;

// Create a MeterProvider with an explicit Exemplar policy
SdkMeterProvider meterProvider = SdkMeterProvider.builder()
    // Other configurations (Readers, Views, etc.) goes here...
    
    // Explicitly set the filter to only record exemplars for traces 
    // that have been selected by the trace sampler.
    .setExemplarFilter(ExemplarFilter.traceBased())
    
    .build();
```

### Backend Support Requirements

Emitting exemplars from your application is only half the battle; your metric infrastructure must be capable of receiving and storing them.

* **Prometheus:** Exemplar storage is supported but often requires explicitly enabling the feature flag (`--enable-feature=exemplar-storage`) depending on the version. Prometheus stores these exemplars in a circular buffer in memory, meaning they are short-lived. They are designed for active incident response, not historical analysis.
* **OTLP Exporters:** Ensure you are using the OpenTelemetry Protocol (OTLP) to export your metrics. Older, legacy exporters (like the Prometheus scrape endpoint in older SDK versions) may strip exemplar data before it leaves the application process.

By combining Synchronous Histograms with Trace-Based Exemplars, observability practitioners achieve the "Holy Grail" of incident response: the ability to detect an anomaly at a macro level and drill down to the exact micro-level root cause in milliseconds, without writing a single manual query.
